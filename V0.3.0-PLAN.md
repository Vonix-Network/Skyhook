# Skyhook v0.3.0 — full SSH + production polish

**Goal:** ship a complete remote-ops desktop client. SFTP + SSH terminal + key management + port forwarding + the polish that v0.2 left on the floor.

## In scope

### SSH terminal (the headline)
- **PTY channel** on the existing SSH session — one or more interactive shells per connection.
- **xterm.js terminal pane** — toggleable bottom panel (VSCode-style) OR new tab type. Subagent picks the cleaner UX.
- **Resize forwarding** — terminal resize → SSH WINDOW_CHANGE.
- **Copy/paste** — Ctrl+Shift+C / Ctrl+Shift+V (Linux/Win), Cmd+C / Cmd+V (macOS).
- **Multiple shells per session** so you can run `tail -f` in one and a deploy in another.
- **Reconnect terminal** when the parent session reconnects.

### Transfer UX (deferred from v0.2)
- **Live progress bars** wired to `transfer-progress` events.
- **Pause / Resume / Cancel** buttons per row in the TransferPanel.
- **Throughput (MB/s)** + **ETA** computed from bytes/total + start time.
- **Clear-completed** button.

### Production polish
- **Window state persistence** (size, position, maximized) — settings schema is already there, just needs wiring on the Rust side.
- **Last-active connection auto-focus** on startup.
- **Empty states designed**, not text — illustration + helpful CTA for empty dirs, empty connection list, no transfers.
- **Properties modal** instead of `alert()` for file details.
- **Tooltips** on toolbar buttons + keyboard shortcuts inline.
- **Keyboard-shortcut help overlay** (press `?`).
- **Resizable sidebar** + **resizable transfer panel** (drag the borders).
- **Connection import/export** as JSON (without secrets — only host/port/user/name).

### v0.3 nice-to-haves (only if budget allows)
- **In-app SSH keygen** (Ed25519 + RSA 4096).
- **Auto-updater** via Tauri's update plugin.
- **Port forwarding** (-L, -R) — really good differentiator vs FileZilla but expensive.

## Out of scope (v0.4+)
- Recursive remote search
- Mount-as-drive (FUSE/Dokan/macFUSE)
- Watch-and-sync folders
- Cloud-synced bookmarks

## Architecture — what changes

### Backend: SSH shell subsystem
Add a peer to the SFTP channel inside `SessionActor`:

```
SessionActor
 ├─ Op queue (existing)              — SFTP ops
 ├─ Shell registry (new)             — HashMap<ShellId, ShellHandle>
 │   └─ ShellHandle
 │       ├─ russh::Channel<Msg>
 │       ├─ tokio::mpsc::Sender for stdin
 │       └─ reads channel data → emits "shell-output" event
 └─ Heartbeat (existing)
```

New commands:
```
shell_open({sessionId, cols, rows}) -> ShellInfo { id }
shell_write({shellId, data})        -> void          // data is plain string
shell_resize({shellId, cols, rows}) -> void
shell_close({shellId})              -> void
```

New events:
```
"shell-output":  { shellId, data: string }   // pre-decoded UTF-8 (xterm handles it)
"shell-closed":  { shellId, exit_code: number | null }
```

### Backend: transfer engine progress
Already emits `transfer-progress`. Just need the frontend to subscribe and render properly. Maybe expose throughput at the actor level instead of computing in JS.

### Frontend: terminal component
xterm + xterm-addon-fit already in `package.json`. New component `Terminal.tsx` mounting a single shell with proper lifecycle.

### Wave plan

Wave 1 (parallel x3, backend-heavy):
- **A** — SSH shell subsystem in `session/shell.rs` + new `shell_*` commands + events.
- **B** — Transfer progress UI rewrite (progress bars, pause/cancel buttons, throughput, ETA).
- **C** — Production polish backend (window state save/restore, last-active connection, properties dialog command, import/export commands, ssh_keygen optional).

Wave 2 (parallel x3, frontend-heavy):
- **D** — Terminal frontend (xterm.js, integration with sessions, multi-shell UX).
- **E** — Polish frontend (resizable sidebar/panel, empty states, tooltips, shortcuts modal, properties modal).
- **F** — Auto-updater integration + connection import/export UI + ssh keygen UI (if backend C shipped it).

Wave 3 (main thread):
- Integration tests.
- CHANGELOG.
- Build + tag.

## Schema contracts (pre-defined)

```ts
// Shell
interface ShellInfo { id: string; session_id: string; }
interface ShellOutputEvent { shell_id: string; data: string; }
interface ShellClosedEvent { shell_id: string; exit_code: number | null; }

// Window state (from existing Settings)
interface WindowState { width?: number; height?: number; x?: number; y?: number; maximized: boolean; }

// Connection export bundle (NO SECRETS)
interface ConnectionExport {
  version: 1;
  connections: Array<{
    name: string; host: string; port: number; username: string;
    default_path?: string; color?: string;
    auth_kind: 'password' | 'key' | 'agent';  // metadata only, no actual secret
  }>;
}
```
