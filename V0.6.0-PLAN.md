# Skyhook v0.6.0 — Agent Integration

> **For Hermes:** Use subagent-driven development. Three waves, parallel x3 each, main-thread integration between waves.

**Goal:** Skyhook ships with an integrated AI agent that drives the existing SFTP + SSH primitives. The agent can read/write/upload/download files, run shell commands, and reason about a remote box — all through the same Tauri session the user already has open. Supports Anthropic and OpenAI with native SDKs and aggressive prompt caching.

## Why this differs from Cursor/Claude Code

- Cursor / Claude Code work on your **local checkout**.
- Skyhook Agent works on a **live remote box** through SSH+SFTP.
- The agent's "filesystem" tools route through `SessionHandle`, not `std::fs`. That's the entire differentiator.

## Tech stack

- Anthropic SDK: `anthropic-sdk` (Rust) or direct `reqwest` with the Messages API. Reqwest direct gives us tighter control over `cache_control` breakpoints.
- OpenAI SDK: `async-openai` crate, OR direct `reqwest` with Chat Completions API. Use Chat Completions for broadest compatibility.
- Streaming: `reqwest` + `eventsource-stream` for SSE.
- Tool dispatch: in-process. Tools are Rust enums dispatched against the existing `SessionHandle` API.
- Persistence: JSON files at `<config>/skyhook/agent/<connection_id>/<conversation_id>.json`.
- Keys: OS keyring (existing `keyring` dep).

## Prompt cache strategy

### Anthropic (4 breakpoints max)
1. **System prompt + tool defs** — cache_control: `{type: "ephemeral"}` — long-lived, never changes per-conversation.
2. **Earliest user turn + assistant reply** — cached after first reply lands.
3. **Sliding window mid-conversation** — re-cache every 5 turns to keep the recent context cheap.
4. **Last assistant message before current user turn** — cached just-in-time.

Total: ~75% input token cost reduction on long conversations.

### OpenAI
- Use chat completions with `prompt_cache_key` (when on a model that supports it: GPT-4o, o1, o3, gpt-4.1+).
- System prompt is the cacheable prefix — keep it static.
- Tool definitions go in the system message body so they're part of the cached prefix.

## Tool surface (what the agent can call)

| Tool | Maps to | Approval? |
|---|---|---|
| `sftp_list_dir(path)` | `SessionHandle::list_dir` | no |
| `sftp_read_file(path, max_bytes?)` | `SessionHandle::read_file` | no |
| `sftp_stat(path)` | `SessionHandle::stat` | no |
| `sftp_walk(root, max_depth?)` | `SessionHandle::walk` | no |
| `sftp_write_file(path, content)` | `SessionHandle::write_file` | **YES** |
| `sftp_make_dir(path)` | `SessionHandle::mkdir` | **YES** |
| `sftp_remove(path)` | `SessionHandle::remove` | **YES** |
| `sftp_rename(from, to)` | `SessionHandle::rename` | **YES** |
| `sftp_download(remote, local)` | `SessionHandle::download` | **YES** |
| `sftp_upload(local, remote)` | `SessionHandle::upload` | **YES** |
| `shell_exec(command, timeout_s?)` | Open ephemeral shell, run, capture, close | **YES** |
| `task_complete(summary)` | Marks end-of-turn | no |

`shell_exec` does NOT use the user's existing shell tabs — it opens a fresh, ephemeral PTY, runs the command non-interactively, captures stdout+stderr+exit_code, closes.

## Approval modes

Per connection (stored in connection profile):
- **Manual**: every write/exec confirmed
- **Auto-read**: reads always allowed, writes/execs confirmed (default)
- **YOLO**: everything goes (requires typed confirmation when enabling)

## Architecture

### Backend `src/agent/`

```
src/agent/
├── mod.rs            — pub re-exports, AgentService struct
├── provider.rs       — trait Provider, with stream_chat() returning Stream<Event>
├── anthropic.rs      — AnthropicProvider impl (Messages API + cache_control)
├── openai.rs         — OpenAIProvider impl (Chat Completions + prompt_cache_key)
├── tools.rs          — Tool enum, tool schemas as JSON, dispatch_tool()

├── approval.rs       — ApprovalGate, ApprovalDecision
├── history.rs        — ConversationStore, message log, JSON persistence
├── keystore.rs       — API key get/set via OS keyring
└── system_prompt.rs  — system prompt builder (cacheable prefix)
```

## System prompt (cacheable prefix)

```
You are Skyhook, an AI assistant integrated into a desktop SFTP+SSH client.
You are connected to a remote server via SSH and SFTP. The current connection is
to {host}:{port} as user {username}. The working directory is {cwd}.

You can:
- Read and write files on the remote server (via SFTP)
- Walk the directory tree
- Run shell commands (via a fresh PTY per command)
- Upload files from the user's local machine to the remote
- Download files from the remote to the user's local machine

Rules:
- Be concise. Long explanations are wasteful when the user is watching tool output.
- For destructive operations, explain your intent in one short sentence before calling the tool.
- Read before writing. Confirm assumptions about file structure before editing.
- When editing a config file, read it first, then write the full new contents.
- When running shell commands, prefer one-shot commands with clear output.
- The user can reject any write/exec. If rejected, ask for clarification rather than retrying.
- When you've completed the user's request, call task_complete with a one-line summary.
```

Plus the tool schemas appended.

## Waves

### Wave 1 — backend (parallel x3)
- **A**: Provider trait, Anthropic impl with cache_control, streaming SSE → events.
- **B**: OpenAI impl (Chat Completions, tool calls, prompt_cache_key), models list endpoint, keystore (existing keyring).
- **C**: Tools enum + dispatch routing to SessionHandle, AgentRunner main loop, ApprovalGate, ConversationStore (JSON file persistence).

### Wave 2 — frontend (parallel x3)
- **D**: agent-store, AgentPanel shell, MessageList + Message + Composer + streaming wiring.
- **E**: ToolCallCard + ApprovalCard + DiffPreview (Monaco diff), tool result rendering.
- **F**: ConversationList + AgentSettings (provider switch, model dropdown, key inputs, mode toggle), per-connection approval-mode override UI.

### Wave 3 — integration + release (main thread)
- Wire commands into `lib.rs`, verify `cargo check`, smoke-test compile.
- CHANGELOG entry, version bump to 0.6.0.
- Release build + push + tag.
